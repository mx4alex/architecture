# ADR-003: Кэширование динамики бюджета в Redis (паттерн Cache-Aside)

**Date:** 2026-03-08

**Status:** Accepted

## Контекст

Расчёт динамики бюджета за период — самая тяжёлая операция в системе: требуется выборка всех доходов и расходов пользователя за указанный интервал, их агрегация и вычисление итогового баланса по дням/неделям/месяцам. При частых обращениях к одному и тому же периоду повторное выполнение этих запросов к PostgreSQL создаёт избыточную нагрузку на СУБД.

Факторы, влияющие на решение:

- Одни и те же периоды запрашиваются многократно — паттерн повторяемости высок.
- Данные бюджета изменяются относительно редко, а читаются значительно чаще.
- Допустима небольшая задержка в отображении обновлённых данных.

## Рассматриваемые варианты

### Вариант 1: Redis с паттерном Cache-Aside и TTL

**Плюсы:**

- In-memory хранилище c временем доступа <1 мс, кардинальное ускорение повторных запросов.
- Cache-Aside простой в реализации паттерн: приложение проверяет кэш, при промахе запрашивает БД и сохраняет результат.
- TTL автоматически инвалидирует устаревшие данные без дополнительной логики.
- Redis поддерживает различные структуры данных, что даёт гибкость при расширении функционала кэширования.

**Минусы:**

- Дополнительный инфраструктурный компонент, требующий мониторинга.
- Данные в кэше могут быть устаревшими в пределах TTL.
- Риск cache stampede при одновременном истечении TTL для популярных ключей.

### Вариант 2: Локальный in-process кэш (LRU в памяти сервиса)

**Плюсы:**

- Нет дополнительных инфраструктурных компонентов.
- Нулевая сетевая задержка, данные находятся в памяти процесса.

**Минусы:**

- Кэш не разделяется между репликами Budget Service, при горизонтальном масштабировании каждая реплика хранит свою копию, что увеличивает потребление памяти и снижает hit rate.
- Инвалидация при обновлении данных усложняется.
- Перезапуск сервиса полностью сбрасывает кэш.

### Вариант 3: Не использовать кэширование

**Плюсы:**

- Простота: нет проблем с инвалидацией и согласованностью.
- Данные всегда актуальны.

**Минусы:**

- Каждый повторный запрос динамики за один и тот же период выполняет полную агрегацию в PostgreSQL.
- При росте числа пользователей нагрузка на СУБД растёт линейно, что приведёт к деградации производительности.
- Потребуется более мощный инстанс PostgreSQL.

## Решение

Мы используем **Redis** для кэширования результатов расчёта динамики бюджета по паттерну **Cache-Aside с TTL**. Ключ кэша формируется как `dynamics:{userId}:{from}:{to}`. При запросе Budget Service сначала проверяет Redis; при промахе выполняет SQL-запрос к PostgreSQL, сохраняет результат в Redis с TTL и возвращает клиенту. При добавлении/изменении дохода или расхода соответствующие ключи инвалидируются явно.

## Последствия

**Положительные:**

- Повторные запросы динамики за популярные периоды обслуживаются из Redis за <1 мс вместо 50–200 мс SQL-агрегации.
- Нагрузка на PostgreSQL снижается пропорционально hit rate кэша.
- При горизонтальном масштабировании Budget Service все реплики разделяют единый кэш.

**Отрицательные:**

- В пределах TTL пользователь может видеть устаревшую динамику после добавления новой записи, если явная инвалидация не сработала. Это допустимо для аналитического представления.
- Дополнительный компонент инфраструктуры, требующий мониторинга.
- При потере данных Redis все запросы пойдут в PostgreSQL, следовательно, будет временная деградация производительности.

**Риски:**

- Риск cache stampede при одновременном истечении TTL для популярных ключей. Снижение: использование случайного jitter к TTL (±30 секунд) для распределения нагрузки.
