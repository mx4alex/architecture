# ADR-002: Выбор PostgreSQL как основной СУБД

**Date:** 2026-03-08

**Status:** Accepted

## Контекст

Системе бюджетирования необходимо хранить данные пользователей (логин, хэш пароля, профиль), планируемые доходы и расходы и выполнять агрегирующие SQL-запросы для расчёта динамики бюджета за произвольный период. Данные имеют реляционную природу: доходы/расходы привязаны к пользователю, имеют категорию, сумму, дату. Целостность финансовых данных критична.

Факторы, влияющие на решение:

- Данные строго структурированы и связаны между собой.
- Требуется поддержка ACID-транзакций для корректного учёта финансовых операций.
- Необходимы эффективные агрегирующие запросы для расчёта динамики.

## Рассматриваемые варианты

### Вариант 1: PostgreSQL

**Плюсы:**

- Полная поддержка ACID-транзакций, гарантия целостности финансовых данных.
- Мощный оптимизатор запросов и оконные функции удобны для аналитических агрегаций.
- Зрелый open-source проект с активным сообществом и предсказуемым развитием.
- Поддержка JSONB для хранения слабоструктурированных данных в будущем.

**Минусы:**

- Горизонтальное масштабирование сложнее, чем у NoSQL-решений.
- Требует настройки для оптимальной работы под нагрузкой.

### Вариант 2: MySQL 

**Плюсы:**

- Широко распространена, большое сообщество.
- Простая настройка репликации.

**Минусы:**

- Оптимизатор запросов слабее справляется со сложными аналитическими выборками.
- Реализация MVCC через undo-логи: при конкурентных чтении и записи длинные аналитические запросы могут вызывать рост undo-сегментов и деградацию производительности.

### Вариант 3: MongoDB

**Плюсы:**

- Гибкая схема данных, простота горизонтального масштабирования.
- Хорошая производительность на операциях записи.

**Минусы:**

- Мульти-документные транзакции работают с заметным overhead (координация, блокировки), тогда как в системе бюджетирования каждая операция затрагивает связанные записи, и транзакционность нужна постоянно.
- Aggregation pipeline значительно сложнее и менее производителен для реляционных по природе запросов по сравнению с SQL, где это решается одним выражением.

## Решение

Мы используем **PostgreSQL** в качестве единой СУБД для хранения данных пользователей, планируемых доходов и расходов. Обоим сервисам (User Service и Budget Service) выделяются отдельные схемы в рамках одного инстанса PostgreSQL для логической изоляции данных при простоте администрирования.

## Последствия

**Положительные:**

- ACID-транзакции гарантируют целостность финансовых данных при конкурентном доступе.
- Оконные функции и CTE позволяют эффективно рассчитывать динамику бюджета на стороне СУБД.

**Отрицательные:**

- Единый инстанс PostgreSQL является точкой отказа. При значительном росте нагрузки потребуется горизонтальное масштабирование.
- Два сервиса используют одну СУБД — формально нарушается принцип database-per-service. Это осознанный компромисс ради простоты администрирования на текущем этапе.
- Необходимо настроить connection pooling при росте числа соединений.

**Риски:**

- Риск взаимного влияния сервисов через общую СУБД (тяжёлый аналитический запрос Budget Service замедляет User Service). Снижение: разделение на схемы, мониторинг медленных запросов, при необходимости вынесение в отдельные инстансы.
