# ADR-001: Использование Kong в качестве API Gateway

**Date:** 2026-03-08

**Status:** Accepted

## Контекст

Система бюджетирования состоит из нескольких бэкенд-сервисов (User Service и Budget Service). Клиентское SPA-приложение не должно знать о внутренней топологии сервисов и напрямую обращаться к каждому из них. Необходим единый входной endpoint, который берёт на себя маршрутизацию, проверку аутентификации и защиту от перегрузки.

Факторы, влияющие на решение:

- Клиенту нужен единый URL для всех API-запросов.
- Проверка JWT-токена должна выполняться централизованно, а не дублироваться в каждом сервисе.
- Система должна быть защищена от DDoS/брутфорс-атак через rate limiting.

## Рассматриваемые варианты

### Вариант 1: Kong (Open Source)

**Плюсы:**

- Зрелый продукт с развитой экосистемой плагинов (JWT, rate-limiting, logging, CORS).
- Высокая производительность благодаря ядру на базе Nginx/OpenResty.
- Декларативная конфигурация, удобная для Infrastructure as Code.
- Активное сообщество, обширная документация.

**Минусы:**

- Дополнительный инфраструктурный компонент, требующий мониторинга.
- Плагинная модель Lua менее привычна команде.
- Enterprise-функции доступны только в платной версии.

### Вариант 2: Nginx как reverse proxy с кастомной конфигурацией

**Плюсы:**

- Хорошо знаком большинству DevOps-инженеров.
- Минимальный overhead — лёгкий и быстрый.

**Минусы:**

- JWT-валидацию, rate limiting и другие cross-cutting concerns придётся реализовывать через модули или Lua-скрипты вручную.
- Нет встроенного управления маршрутами к сервисам — конфигурация через статические файлы, менее гибка.
- Отсутствует экосистема готовых плагинов для API-задач.

### Вариант 3: Кастомный API Gateway на C++

**Плюсы:**

- Единый стек технологий с остальными сервисами.
- Полный контроль над логикой маршрутизации.

**Минусы:**

- Значительные трудозатраты на реализацию и поддержку: JWT-валидация, rate limiting, логирование, health checks.
- Отвлекает команду от бизнес-логики.
- Высокий риск ошибок в security-критичном коде.

## Решение

Мы используем **Kong (Open Source)** в DB-less режиме в качестве API Gateway. Kong берёт на себя маршрутизацию запросов к User Service (`/users/*`) и Budget Service (`/budget/*`), JWT-валидацию и rate limiting. Конфигурация хранится в YAML-файле и версионируется в Git.

## Последствия

**Положительные:**

- SPA-клиент работает с единым URL, внутренняя топология сервисов скрыта.
- Централизованная проверка JWT избавляет сервисы от дублирования логики аутентификации.
- Rate limiting защищает сервисы от перегрузки без дополнительного кода.
- Декларативная конфигурация упрощает воспроизводимость окружения.

**Отрицательные:**

- Дополнительный компонент в инфраструктуре, требующий мониторинга и обновлений.
- Все запросы проходят через дополнительный сетевой hop, что добавляет несколько миллисекунд латентности.
- Команде необходимо освоить конфигурацию Kong и написание Lua-плагинов при расширении логики.

**Риски:**

- Риск Kong стать single point of failure. Снижение: запуск минимум двух реплик Kong за балансировщиком нагрузки, настройка health checks.
